# Crossplane - Abstract an application and infrastructure in a composition

**Demonstrate**:
* How to innstall and configure provider-vault, rovider-sql and External Secrets Operator
* How to use namespaced apis "m." of providers
* How to abstract a product line called `RazorApps` with XRD and Composition (KCL)
* How to instanciate different projects of this product line by a simple XR
* How to read secrets back from Vault and mount it in the deployment

**Prerequisites**:
* Crossplane is setup in your cluster (see [Installation](../Readme.md#install-helm-chart-in-kind-cluster))
* Vault is setup in your cluster incl. port-forwarding (see [Installation](../../vault/Readme.md#install-helm-chart-in-kind-cluster))
* Vault CLI is configured (see [Installation](../../vault/Readme.md#usage))

## Prepare environment

### External Secrets Operator

```bash
# Install external secrets operator
# to make use of the pushSecret to safe the secret generated by the provider-sql to Vault
helm repo add external-secrets https://charts.external-secrets.io
helm repo update
# Installation
helm install external-secrets external-secrets/external-secrets \
    --namespace external-secrets \
    --create-namespace \
    --set installCRDs=true

# Verify that ESO is running (cert and webhook can take up to 2m30sec)
kubectl -n external-secrets get pods --watch
```

### Vault
Prepare Vault resources our product line will build on
```bash
# vault login (with root token)
export VAULT_ADDR='http://localhost:8200'
vault login

# Enable kubernetes authentication
vault auth enable --path=kubernetes kubernetes

# Configure kubernetes auth backend
vault write auth/kubernetes/config \
    kubernetes_host=https://kubernetes.default.svc.cluster.local

# Create secret engine that will be used for the projects of our product line
vault secrets enable -path=projects kv-v2
```

### Local MSSQL server
```bash
# Start local mssql server with docker-compose
docker-compose up -d

# Connect mssql docker container to docker network of kind
# Special Step for Kind on Linux, Kind clusters often run on their own Docker network (named kind).
# If your MSSQL container is on the default bridge network, they might not see each other.
# Connect your MSSQL container to the Kind network: 

docker network connect kind mssql-test
# After connecting, your MSSQL container can be reached by its container name (e.g., mssql-test) from inside the cluster.
```

### Crossplane - Install and configure providers and functions
Install Vault and SQL provider and function to write KCL pipelines
```bash
# Install Crossplane Vault provider
kubectl apply -f 1-crossplane-provider-and-function-install.yaml

# Check provider and function pods are running
kubectl get pods -n crossplane-system --watch

# Configure
kubectl apply -f 2-crossplane-provider-configuration.yaml
```

### External Secrets Operator
Enable crossplane through RBAC to manage ESO objects and configure "global" secret store
```bash
# Configure External Secrets Operator SecretStore to be used for database credentials by provider-sql
kubectl apply -f 3-eso-rbac-crossplane-and-eso-config.yaml
```

### Register XRD and Composition for our product line RazorApp
```bash
kubectl apply -f 4-app-abstraction-xrd-comp.yaml

# Verify new CRD is available
kubectl get crd | grep razor
```

## Deploy applications
### Deploy the Blade example app

Now, as the basic infrastructure and platform is setup, let's create 2 example apps that make use of it. The applications create:
* Application Service account and Deployment
* MSSQL database, users and grants
* PushSecrets to store db secrets in Vault
* Vault policy and Kubernethes auth role for instanciated app to read its own credentials from Vault
* ESO SecretStore and ExternalSecret to read back the credentials from Vault to create a secret in the application namespace to consume it

```bash
# Create the razor application
kubectl create ns blade
kubectl apply -f 5-xr-blade.yaml

# Let's explore if our objects were created

# Get database and users
kubectl -n blade get databases.mssql.sql.m.crossplane.io
kubectl -n blade get users.mssql.sql.m.crossplane.io

# Verify local db secrets exist (created by provider-sql) and is not empty
k -n blade get secret

Output:
blade-readonly-user-secret    connection.crossplane.io/v1alpha1   4      12m
blade-readwrite-user-secret   connection.crossplane.io/v1alpha1   4      12m

# Check status of pushSecrets (Status should be Synced)
kubectl -n blade get pushsecrets.external-secrets.io

# Optional if needed: Manually trigger a sync of an ESO pushSecret
kubectl -n blade annotate pushsecret push-razor-db-ro-user-to-vault force-sync=$(date +%s) --overwrite

# Retrieve the credentials of the database readonly user
vault kv get projects/blade/database/ro-user

# Retrieve credentials and try to read sys tables (should work)
DB_ENDPOINT=$(vault kv get -field=endpoint projects/blade/database/ro-user)
DB_PORT=$(vault kv get -field=port projects/blade/database/ro-user)
RO_USER=$(vault kv get -field=username projects/blade/database/ro-user)
RO_PW=$(vault kv get -field=password projects/blade/database/ro-user)

docker exec -it $DB_ENDPOINT /opt/mssql-tools18/bin/sqlcmd -S localhost,$DB_PORT -U $RO_USER -P $RO_PW -C -Q "SELECT name FROM sys.tables"

# Output:
name
--------------------------------------------------------------------------------------------------------------------------------
spt_fallback_db
spt_fallback_dev
spt_fallback_usg
spt_monitor

(4 rows affected)

# Let's check the kubernetes auth role
vault list auth/kubernetes/role

# Output:
Keys
----
blade-app

# Let's check our application policies
# If application is only allowed to push secrets to its own project
vault policy read blade-eso-push-policy

# Output:
path "projects/data/blade/*" {
    capabilities = ["create", "update", "read", "delete", "list"]
}
path "projects/metadata/blade/*" {
    capabilities = ["create", "update", "read", "delete", "list"]
}

# if we only have access to the application secrets when reading
vault policy read blade-app-read-policy

# Output:
path "projects/data/blade/*" {
    capabilities = ["read", "list"]
}

# Check the secret created by the ExternalSecret that reads the credentials back from Vault
# and makes it available for the application to consume

# Check deployment
kubectl -n blade get all

# Verify the secret is synced
kubectl -n blade get externalsecrets.external-secrets.io

# Verify that secrets read from Vault are created
kubectl -n blade get secret | grep database

# See if it contains all 4 attributes
kubectl -n blade get secret database-user-ro -o jsonpath='{.data}'

# Verify if the secret contents are loaded successully in the environment of our application pod
kubectl -n blade exec deployments/frontend -- printenv | grep -E "endpoint|port|username|password"
```
