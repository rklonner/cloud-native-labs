# Crossplane - Test various Vault integrations

**Demonstrate**:
* How to configure Vault provider
* How to create secret engines and secrets 
* How to create kubernetes auth role and policy for registering an application
* How to utilize External Secrets operator pushSecrets to store credentials of provider-sql database
* How to grant access to existing secrets

**Prerequisites**:
* Crossplane is setup in your cluster (see [Installation](../Readme.md#install-helm-chart-in-kind-cluster))
* Vault is setup in your cluster incl. port-forwarding (see [Installation](../../vault/Readme.md#install-helm-chart-in-kind-cluster))
* Vault CLI is configured (see [Installation](../../vault/Readme.md#usage))

## Prepare environment

### External Secrets Operator

```bash
# Install external secrets operator
# to make use of the pushSecret to safe the secret generated by the provider-sql to Vault
helm repo add external-secrets https://charts.external-secrets.io
helm repo update
# Installation
helm install external-secrets external-secrets/external-secrets \
    --namespace external-secrets \
    --create-namespace \
    --set installCRDs=true

# Verify that ESO is running
kubectl -n external-secrets get pods --watch
```

### Vault
```bash
# vault login (with root token)
export VAULT_ADDR='http://localhost:8200'
vault login

# Enable kubernetes authentication
vault auth enable --path=kubernetes kubernetes

# Configure kubernetes auth backend
vault write auth/kubernetes/config \
    kubernetes_host=https://kubernetes.default.svc.cluster.local

# Create secret engine that will be used for the projects of our product line
vault secrets enable -path=projects kv-v2
```

### Crossplane Vault Provider
```bash
# Install Crossplane Vault provider
kubectl apply -f 1-vault-provider-install.yaml

# Verify the provider is running
kubectl get pods -n crossplane-system --watch
kubectl get providers.pkg.crossplane.io provider-vault

# Configure the provider
kubectl apply -f 2-vault-provider-configure.yaml
```

### External Secrets Operator
```bash
# Configure External Secrets Operator SecretStore to be used for database credentials by provider-sql
kubectl apply -f 3-eso-config.yaml

# Verify that the SecretStore is healhy
kubectl -n crossplane-system get secretstores.external-secrets.io eso-secret-store-vault-backend
```

### MSSQL database
```bash
# Start local mssql server with docker-compose
docker-compose up -d

# Connect mssql docker container to docker network of kind
# Special Step for Kind on Linux, Kind clusters often run on their own Docker network (named kind).
# If your MSSQL container is on the default bridge network, they might not see each other.
# Connect your MSSQL container to the Kind network: 

docker network connect kind mssql-test
# After connecting, your MSSQL container can be reached by its container name (e.g., mssql-test) from inside the cluster.

# Install the Crossplane provider-sql which supports mssql
kubectl apply -f 4-managed-resources-mssql-install.yaml

# Verify the provider is running
kubectl get pods -n crossplane-system --watch
kubectl get providers.pkg.crossplane.io provider-sql

# Configure the provider
kubectl apply -f 5-managed-resources-mssql-config.yaml
```

### Deploy the Razor example app

Now, as the basic infrastructure and platform is setup, let's create 2 example apps that make use of it. The applications create:
* namespace
* service account
* MSSQL database, db-users
* PushSecrets to store db secrets in Vault
* Vault policy for Razor app to read its own credentials
* Vault Kubernethes auth role for the app
* ESO SecretStore and ExternalSecret to read back the credentials from Vault to create a secret in the application namespace to consume it

```bash
# Create the razor application
kubectl apply -f 6-razor-app.yaml

# Let's explore if our objects were created

# Retrieve the credentials of the database readonly user
vault kv get projects/razor/database/ro-user

# Retrieve credentials and try to read sys tables (should work)
DB_ENDPOINT=$(vault kv get -field=endpoint projects/razor/database/ro-user)
DB_PORT=$(vault kv get -field=port projects/razor/database/ro-user)
RO_USER=$(vault kv get -field=username projects/razor/database/ro-user)
RO_PW=$(vault kv get -field=password projects/razor/database/ro-user)

docker exec -it $DB_ENDPOINT /opt/mssql-tools18/bin/sqlcmd -S localhost,$DB_PORT -U $RO_USER -P $RO_PW -C -Q "SELECT name FROM sys.tables"

# Output:
name
--------------------------------------------------------------------------------------------------------------------------------
spt_fallback_db
spt_fallback_dev
spt_fallback_usg
spt_monitor

(4 rows affected)

# Let's check the kubernetes auth role
vault list auth/kubernetes/role

# Output:
Keys
----
razor-role

# Let's check our application policy if we only have access to the application secrets
vault policy read razor-policy

# Output:
path "projects/data/razor/*" {
  capabilities = ["read", "list"]
}

# Check the secret created by the ExternalSecret that reads the credentials back from Vault
# and makes it available for the application to consume

# Verify the secret is synced
kubectl -n razor describe externalsecrets.external-secrets.io razor-db-credentials-ro

# See if it contains all 4 attributes
kubectl -n razor get secret database-user-ro -o jsonpath='{.data}'

# Verify if the secret contents are loaded successully in the environment of our application pod
kubectl -n razor exec deployments/razor-deployment -- printenv | grep -E "endpoint|port|username|password"
```
